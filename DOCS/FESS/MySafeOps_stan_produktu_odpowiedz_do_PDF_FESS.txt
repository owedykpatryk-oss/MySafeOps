4. STACK I HOSTING (FAKTY)
--------------------------------------------------------------------------------
- Frontend: React (Vite), SPA, react-router. Build: „npm run build”, output dist/.
- Część publiczna i /app: przeglądarka; dane rejestrów w localStorage
  (patrz orgStorage) chyba że funkcja używa innej ścieżki.
- supabase-js: projekt łączy się z Supabase, gdy podane VITE_SUPABASE_*
  (anon key w bundlu — design Vite).
- Stripe: klucz publikowany tylko publishable w UI; logika tajnych kluczy
  po stronie serwera/Edge (nie w tym pliku ujemnym — patrz README/SECURITY).
- Vercel: opis w vercel.json — m.in. nagłówki HSTS, X-Frame-Options, CSP, cache
  /api/* = no-store, /assets długi cache, SPA rewrite.
- public/_headers: odpowiednik nagłówków dla hostów typu Cloudflare Pages
  (SECURITY.md mówi o synchronizacji z Vercel).
- public/.well-known/security.txt: kontakt mailto (faktyczny adres w pliku
  w repo).
- API Vercel (katalog api/): co najmniej api/web-vitals.js (limity ciała JSON,
  413 gdy przekroczenie), api/anthropic-messages.js (limit 512k JSON, whitelista
  kluczy ciała przekazywanych do Anthropic, sekret opcjonalny).
- Service worker: public/service-worker.js, wersja w SW_VERSION; offline shell.

5. ZABEZPIECZENIA (FAKTY Z KODU — NIE = „Certyfikat ukończony”)
--------------------------------------------------------------------------------
5.1. Transport i nagłówki
     - HSTS, nosniff, X-Frame-Options SAMEORIGIN, Referrer-Policy, Permissions-
       Policy, CSP w vercel.json (długa allowlista connect-src m.in. Supabase,
       Stripe, mapy, ewent. AI, pogoda).
     - Odpowiedzi /api: no-store, nosniff.
     - Treść: SECURITY.md — CSP report-only w _headers; promocja do pełnej CSP
       z nonce jest „do decyzji przy CDN”.

5.2. Aplikacja
     - safeInternalPath: redirect po logowaniu tylko do ścieżek wewnętrznych
       (open redirect ograniczany) — src/utils/safeUrl.js.
     - Hasła w UI: minimalna długość 12 znaków w polityce klienckiej
       (src/config/authPolicy.js); Supabase ma ostateczne reguły serwera.
     - Sentry: opcjonalne VITE_SENTRY_DSN; RouteErrorBoundary może
       Sentry.captureException.
     - MFA: getRequiresMfaStep, MfaLoginChallenge, mfaAal (logika poziomu AAL) —
       zależne od konfiguracji Supabase użytkownika/tenant.
     - Konto platform owner: VITE_PLATFORM_OWNER_EMAIL + migracje
       supabase/…platform_owner_email_allowlist (RPC po stronie DB) — wymagana
       spójność z Supabase.
     - Web vitals: POST z path w payload; fetch credentials same-origin.

5.3. Czego KOD NIE robi sam
     - RLS, polityka haseł, MFA, rate limit w Supabase: konfiguracja w
       dashboardzie.
     - TLS, HSTS na innych hostach: operator infrastruktury.
     - Cyber Essentials / ISO: proces organizacyjny, nie plik w repo.
